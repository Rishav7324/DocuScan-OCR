1. Our Commitment to Accessibility

DocuScan OCR is committed to making its application accessible to all users, including those with disabilities. We recognize that document scanning and OCR are essential tools that should be usable by everyone, regardless of ability.

Because DocuScan OCR is maintained by an individual developer, accessibility efforts are ongoing and iterative. We welcome feedback from the community to help us improve.

2. Conformance Status

The Web Content Accessibility Guidelines (WCAG) 2.2 define requirements for designers and developers to improve accessibility for people with disabilities. However, WCAG is primarily designed for web content, not native Android applications. For Android apps, Google's Accessibility Scanner recommendations and the Android Accessibility Suite guidelines provide the relevant framework.

DocuScan OCR targets partial conformance with recognized Android accessibility best practices. We have not conducted a formal WCAG audit as the application is a native Android app rather than a web application. Instead, we follow Android's built-in accessibility APIs and conventions.

4. Known Limitations

We acknowledge the following accessibility limitations in the current version (2.0.0):

  1. Camera viewfinder. The camera PreviewView is a live camera feed without accessibility labels. Users who rely on TalkBack may need assistance aiming the camera at documents. We recommend using the Gallery import feature as an alternative to camera capture.
  2. Custom swipe gestures. Swipe-to-delete in the document list may not be accessible via TalkBack. We provide a long-press context menu as an alternative action for delete operations, but this may not be discoverable.
  3. PDF/DOCX/TXT export. The exported files inherit the accessibility properties of the source document images, which are raster images without embedded text layers in PDF export (text is overlaid as an invisible layer, but the underlying image may not be accessible).
  4. OCR text selection. The OCR results are displayed as plain text, which is accessible. However, the text selection and copy interaction may not be as refined as with a native text view in all custom composable layouts.
  5. No voice control integration. The app does not implement custom Android Voice Access actions or shortcuts, so users relying entirely on voice control may find some interactions difficult.
  6. No reduced motion option. The app's animations (transitions, FAB, dialogs) follow Material Design 3 defaults and do not provide a dedicated reduced-motion toggle. The app respects the system-level "Remove animations" setting.

These limitations are documented here transparently. As the app is developed by an individual maintainer, we prioritize accessibility improvements based on user feedback and available resources. Feedback on accessibility barriers is warmly welcomed.

5. Testing Approach

Accessibility testing is performed using:

  • Android Accessibility Scanner — a static analysis tool that checks for common accessibility issues at runtime.
  • TalkBack — manual end-to-end navigation testing with TalkBack enabled on reference devices (Pixel 6a, API 33; Moto G Power, API 30).
  • Contrast analysis — manual verification against WCAG AA ratios using the Material Design 3 color palette as a baseline.

We do not currently have the resources to conduct usability testing with users with disabilities, but we welcome feedback from users who rely on accessibility technologies.
