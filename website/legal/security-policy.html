1. Overview

This document describes the security architecture of DocuScan OCR for the benefit of users, security researchers, and anyone evaluating the App's security posture. It also provides guidelines for the responsible disclosure of security vulnerabilities.

The developer encourages independent security review of the App. The source code is available for inspection at the official DocuScan OCR channels.

2. Encryption Architecture

The App uses a layered encryption model combining the Android Keystore, PBKDF2 key derivation, and AES-256-GCM symmetric encryption.

2.1 Key Hierarchy

Android Keystore (hardware-backed or TEE-software, never exported)
    │
    └── Master AES-256-GCM key (generated by KeyGenerator in Keystore)
            │
            ├── Purpose: encrypt/decrypt folder PIN verification
            │            marker ("VERIFIED" string).
            │
            └── (Indirectly via PBKDF2) File encryption keys
                     │
                     └── PBKDF2(user PIN + random 16-byte salt, 100,000 iters)
                              │
                              └── AES-256-GCM key per file
                                       │
                                       └── Encrypts: page image bytes
                                            Format: [12-byte IV][ciphertext][16-byte GCM tag]

2.2 Key Derivation Parameters

ParameterValueRationale
AlgorithmPBKDF2WithHmacSHA256NIST-approved, hardware-accelerated on ARMv8 via Cryptography Extensions
Iterations100,000Balance between security (rate-limiting PIN brute force to ~100ms per attempt) and unlock latency on older devices (minSdk 24 devices from 2016)
Salt16 bytes random (SecureRandom)Ensures identical PINs produce different keys in different folders
Key size256 bitsRequired for AES-256; full output of SHA-256-based PBKDF2
StorageSalt stored alongside encrypted blob in Room databaseNeeded for re-derivation; not secret but increases attack complexity per folder

2.3 GCM Encryption Details

ParameterValue
AlgorithmAES/GCM/NoPadding
Key size256 bits
IV nonce length12 bytes (96 bits, recommended by NIST for GCM)
Authentication tag length128 bits (16 bytes)
IV generationRandom, generated by Cipher.init(ENCRYPT_MODE)
Per-encryption IV uniquenessGuaranteed by random generation (collision probability negligible for the data volumes involved)

2.4 PIN Verification Protocol

  1. User creates a 4-digit PIN for a private folder.
  2. App generates a random 16-byte salt via SecureRandom.
  3. App derives a 256-bit AES key via PBKDF2(PIN, salt, 100,000 iterations).
  4. App encrypts the UTF-8 bytes of the string "VERIFIED" using AES-256-GCM with the derived key.
  5. The encrypted blob (IV + ciphertext + tag) and the salt are stored as pinHash and salt on the folder row in Room.
  6. On verification: re-derive the key from the entered PIN and stored salt, attempt GCM decryption of the stored blob. If the GCM authentication tag is valid and the plaintext equals "VERIFIED", the PIN is correct.
  7. If GCM decryption throws an AEADBadTagException, the PIN is incorrect.

This protocol ensures that the PIN itself is never stored and cannot be recovered from the database. An attacker with full database access must brute-force the PIN through 100,000 PBKDF2 iterations per attempt, rate-limited by the CPU.

3. Threat Model

3.1 Assets Protected

  • Document images — photographs of documents stored in private folders.
  • OCR text — extracted text from documents in private folders.
  • Folder PIN — the 4-digit PIN granting access to private folders.

3.2 Trust Assumptions

  • The Android operating system's sandbox is trusted to isolate the App's data from other applications on the device.
  • The Android Keystore implementation on the device is trusted to protect the master key from extraction.
  • The device's lock screen (PIN, pattern, password, or biometric) is trusted to protect the device against unauthorized physical access.
  • The user is trusted to choose a PIN that is not trivially guessable (e.g., "1234", "0000") and to keep it confidential.

3.3 Threat Scenarios & Mitigations

ThreatScenarioMitigationResidual Risk Level
APK decompilation Attacker decompiles the APK with jadx/apktool to extract source code, keys, or algorithms. Encryption keys never enter source code — they are derived at runtime from PIN + Keystore. The algorithm is public (Kerckhoffs's principle). Low — attacker gains no keys; knowledge of the algorithm does not help without the PIN and Keystore.
SQLite database extraction Attacker with root or ADB access copies the Room database file. Folder PIN stored only as encrypted "VERIFIED" marker. Files encrypted at rest. Plaintext document names and folder structure are visible. Medium — folder structure and document metadata are exposed, but document content (images and OCR text) is encrypted.
Keystore key extraction Attacker attempts to read the AES master key from the Android Keystore. On API 28+ devices with hardware-backed Keystore, key material never leaves the secure processor. On API 24-27 (software-only Keystore), extraction may be possible with physical access. Low (API 28+); Medium (API 24-27).
PIN brute force against database Attacker extracts the encrypted "VERIFIED" blob and salt, then tries all 10,000 PINs offline. PBKDF2 at 100k iterations per attempt — ~16 minutes for full 4-digit space at ~100ms/attempt on modern hardware. Keystore rate-limiting on API 30+ adds additional practical delay. Low — 16 minutes is a practical obstacle for a phone thief; a dedicated attacker with server-grade hardware could reduce this to seconds.
Memory dump Attacker with root access dumps the app process heap to read activeFolderPin. PIN is cleared on app background (onPause). During use, PIN lives in Kotlin StateFlow memory — vulnerable to root-level memory inspection. Medium — requires both root and physical access to an unlocked device while the app is actively accessing a private folder.
OAuth token theft from SharedPreferences Attacker reads sync_prefs.xml containing Google Drive or Dropbox tokens. Tokens are plaintext in SharedPreferences (an accepted architectural decision — see Privacy Policy §8). Tokens are scoped and revocable. Medium-high — mitigate by revoking tokens from provider dashboard if device is compromised.
Cloud provider breach Cloudflare R2, Google Drive, or Dropbox suffers a data breach exposing uploaded documents. Out of developer's control. Users should evaluate provider security before enabling cloud sync. Encryption of documents before upload is not currently supported (uploaded images are plain JPEGs — plan to add client-side encryption for cloud copies is noted). Variable — depends on provider. Uploaded images are unencrypted at rest in the cloud provider's storage. Users should consider this when deciding what to sync.
Man-in-the-middle on cloud upload Attacker intercepts HTTPS traffic during cloud sync. All uploads use HTTPS/TLS 1.2+. R2 uploads additionally use AWS SigV4 request signing for integrity verification. Low — standard TLS protections apply; the threat model assumes TLS is not broken at a protocol level.
Supply chain (malicious dependency) A third-party library used by the App contains a vulnerability or backdoor. Minimal dependencies (7 runtime libraries), all from reputable sources (Google/Maven, JetBrains, Square). Source-available. Standard transitive dependency scanning in Gradle. Low — mitigated by minimal attack surface; not immune to zero-day vulnerabilities in dependencies.

4. Known Security Limitations

The following security limitations are known and documented:

  1. OAuth tokens stored in plaintext. Access tokens and refresh tokens for Google Drive and Dropbox are stored in SharedPreferences without encryption. The risk is partially mitigated because tokens are scoped to minimal permissions and are revocable by the user from the provider's security dashboard. A future update should encrypt tokens using the same Keystore-backed mechanism used for folders.
  2. PBKDF2 iteration count below OWASP recommendation. OWASP 2023 recommends 600,000 iterations for PBKDF2-HMAC-SHA256. This App uses 100,000 iterations to maintain acceptable unlock latency on minSdk 24 devices. This should be increased when minSdk is raised to 31+.
  3. Exported files are unencrypted. PDF, DOCX, and TXT exports are saved to external storage as plain, unencrypted files. The export directory is accessible to other apps with storage permissions. Users should handle exported files with appropriate care.
  4. No remote wipe capability. If a device is lost or stolen, there is no mechanism to remotely delete the App's data. The Android Device Manager's factory reset is the only remote option, and it wipes the entire device, not just the App's data.
  5. No passphrase strength enforcement. The App accepts any 4-digit PIN without checking for common patterns ("1234", "0000", "2580", etc.). Users are responsible for choosing a PIN that is not trivially guessable.
  6. Cloud copies are not client-side encrypted. Document images uploaded to R2, Google Drive, or Dropbox are in their original image format (JPEG) without client-side encryption. They are protected by the cloud provider's own encryption-at-rest but are accessible to the provider and could be subject to legal requests.

5. Responsible Disclosure Guidelines

The developer welcomes security research and vulnerability disclosures. If you believe you have found a security vulnerability in DocuScan OCR, please follow these guidelines to report it responsibly.

5.1 Reporting Process

  1. Do not disclose the vulnerability publicly (including on our support channels, social media, or mailing lists) until the developer has had reasonable time to investigate and address it.
  2. Email your findings to support@prontly.in with the subject line "Security Vulnerability: [brief description]".
  3. Include in your report:
    • A clear description of the vulnerability and its potential impact.
    • Steps to reproduce the vulnerability, including any specific device/OS version where it was observed.
    • Your suggested remediation or mitigation, if any.
    • Your preferred name or pseudonym for acknowledgment (if desired).
  4. The developer will acknowledge receipt within 72 hours and will provide an initial assessment within 7 days.
  5. The developer will work with you to understand and address the issue. The developer will keep you informed of progress and will coordinate disclosure timing.

5.2 What We Commit To

  • Acknowledgment of receipt within 72 hours.
  • A fair and timely investigation of the reported issue.
  • Credit in the release notes and/or a SECURITY.md acknowledgment file (unless you prefer to remain anonymous).
  • Reasonable efforts to fix confirmed vulnerabilities in a timely manner, considering the App's nature as a project maintained by an individual developer.

6. Scope & Out of Scope

6.1 In Scope

  • The App's encryption implementation (Keystore, PBKDF2, AES-GCM).
  • The App's OAuth implementation (PKCE flow, token handling).
  • The App's local data storage (Room database, SharedPreferences, file system).
  • The App's network communication (cloud sync HTTPS requests, OAuth exchanges).
  • The App's authentication mechanism (PIN verification, biometric integration).
  • The App's audit log integrity.

6.2 Out of Scope

  • Vulnerabilities in third-party libraries that are already documented by their maintainers (report those to the library maintainers directly).
  • Vulnerabilities in the Android operating system or Google Play Services (report those to Google's Android Security team).
  • Vulnerabilities in Google Drive, Dropbox, or Cloudflare R2 (report those to the respective providers).
  • Social engineering attacks against the developer or users of the App.
  • Physical attacks requiring prolonged unattended access to an unlocked device.
  • Denial of service attacks against the App or any associated infrastructure.
  • Theoretical attacks that require impractical resources (e.g., breaking AES-256 with quantum computers).
  • Self-XSS or vulnerabilities that require the attacker to already have full control of the target device.

6.3 Safe Harbor

The developer will not pursue legal action against individuals who:

  • Engage in security research conducted in good faith, following the disclosure process described above.
  • Limit their testing to their own device(s) and do not access or attempt to access other users' data without their consent.
  • Do not destroy, corrupt, or exfiltrate data beyond what is necessary to demonstrate the vulnerability.
  • Do not publicly disclose the vulnerability before the developer has had a reasonable opportunity to address it.

Security research conducted in accordance with this policy is considered authorized access and is protected under safe harbor provisions of applicable computer fraud and abuse laws.
